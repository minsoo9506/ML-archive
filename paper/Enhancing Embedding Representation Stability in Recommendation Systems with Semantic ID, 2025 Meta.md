# Enhancing Embedding Representation Stability in Recommendation Systems with Semantic ID

- **저자**: Carolina Zheng et al. (Meta, 18명)
- **연도**: 2025
- **링크**: https://arxiv.org/abs/2504.02137

---

## 1. 문제 정의

산업 규모 추천 시스템에서 ID 기반 모델이 겪는 3가지 구조적 문제:

| 문제 | 설명 |
|------|------|
| **Item Cardinality** | 전체 아이템 수(I) ≫ 임베딩 테이블 크기(H) → 충돌 불가피 |
| **Impression Skew** | 상위 0.1% 아이템이 전체 노출의 25% 차지 → tail 아이템 학습 부족 |
| **ID Drifting** | 6일 후 원본 코퍼스의 50%가 퇴출, 새 아이템 진입 → 임베딩 drift 발생 |

**Random Hashing의 한계**: 무작위 충돌로 서로 다른 아이템이 동일 임베딩을 공유 → 데이터 오염, 의미 있는 지식 이전 불가

## 2. 방법론
- 어차피 H < I 라서 충돌은 피할 수 없음
- 그렇다면 충돌 방식을 설계하자
- RQ-VAE로 의미 구조를 파악하고, Prefix-ngram으로 의미적으로 유사한 것끼리 충돌시키면 Random Hashing보다 훨씬 좋은 표현 학습 가능

```
동일한 제약 조건: 임베딩 테이블 슬롯 H개 (메모리 한계)
                        │
              ┌─────────┴─────────┐
              │                   │
    [Random Hashing]       [Semantic ID Prefix-ngram]
              │                   │
    무작위 충돌             의미 있는 충돌
    지식 이전 X             지식 이전 O
              │                   │
           성능 낮음           성능 높음
```

### 2.1 기본 모델 구조 (DLRM 기반)

- 임베딩 테이블 **E** ∈ ℝ^(H×d_m)
- 원본 ID x에 대한 임베딩: **e_f(x)** = Σ **e_fi(x)** (합 풀링)
- 평가 메트릭: **Normalized Entropy (NE)** — 낮을수록 좋음

```
                    ┌─────────────────────────────────────────────┐
                    │           DLRM Architecture                 │
                    └─────────────────────────────────────────────┘

  [Sparse Features]        [Dense Features]       [User History]
  (Item ID, User ID, ...)   (age, price, ...)    (clicked items, ...)
        │                         │                      │
        ▼                         ▼                      ▼
 ┌─────────────┐          ┌──────────────┐      ┌───────────────────┐
 │  Embedding  │          │   Bottom MLP │      │ Aggregation Module│
 │    Table    │          │  (FC layers) │      │ (Transformer/PMA) │
 │  E ∈ R^H×d │          └──────┬───────┘      └────────┬──────────┘
 └──────┬──────┘                 │                       │
        │ e_f(x) = Σ e_fi(x)    │                       │
        ▼                        ▼                       ▼
   dense vector             dense vector            dense vector
        │                        │                       │
        └────────────────────────┴───────────────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │   Feature Interaction   │
                    │  dot(vᵢ, vⱼ) for all   │
                    │     pairs (i, j)        │
                    └────────────┬────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │       Top MLP           │
                    │     (FC layers)         │
                    └────────────┬────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │   sigmoid(logit) → p    │
                    │   (click probability)   │
                    └─────────────────────────┘
```

### 2.2 Semantic ID 생성 (2단계)

**전체 파이프라인:**

```
  광고 아이템 (텍스트 + 이미지 + 비디오)
              │
              ▼
  ┌───────────────────────────┐
  │   Content Understanding  │  ← 멀티모달 사전학습 모델
  │        Model (CU)        │    (CC100 데이터 + 내부 광고 데이터 fine-tuning)
  └───────────────────────────┘
              │
              ▼
        z ∈ ℝ^d                ← 밀집 콘텐츠 임베딩 (연속값)
              │
              ▼
  ┌───────────────────────────┐
  │         RQ-VAE            │  ← 오프라인 훈련 후 고정
  │  (Residual Quantization)  │
  └───────────────────────────┘
              │
              ▼
   c = (c₁, c₂, c₃, c₄, c₅, c₆)  ← Semantic ID (이산 코드 시퀀스)
```

**RQ-VAE 내부 동작**: 연속 임베딩 z를 계층적으로 근사, 각 레이어에서 잔차(residual)를 줄여나감

```
입력: z (콘텐츠 임베딩)

Layer 1:  r₁ = z
          c₁ = arg min ||codebook₁[c] - r₁||₂
          approximation₁ = codebook₁[c₁]

Layer 2:  r₂ = z - approximation₁        ← 1층 근사 후 남은 잔차
          c₂ = arg min ||codebook₂[c] - r₂||₂
          approximation₂ = approximation₁ + codebook₂[c₂]

Layer 3:  r₃ = z - approximation₂        ← 2층 근사 후 남은 잔차
          c₃ = arg min ||codebook₃[c] - r₃||₂
          ...

최종 Semantic ID: c = (c₁, c₂, c₃, c₄, c₅, c₆)
                      각 cᵢ ∈ {0, 1, ..., 2047}  (K=2048, codebook 마다 2048 개 클러스터라고 생각가능)
```

**계층적 의미 구조**: 상위 레이어일수록 거친 의미(카테고리), 하위 레이어일수록 세부 속성을 담음

```
c₁             → "스포츠 용품" 카테고리
c₁, c₂         → "스포츠 용품 > 러닝화"
c₁, c₂, c₃     → "스포츠 용품 > 러닝화 > 경량 쿠셔닝형"
...
c₁ ~ c₆        → 가장 세밀한 의미 표현
```

**RQ-VAE 수식:**
```
r_l = z - Σᵢ₌₁^(l-1) v^i_{c_i}
c_l = arg min_c ||v^l_c - r_l||₂
```

프로덕션 설정: L=6, K=2048, 훈련 데이터 = 최근 3개월 광고 샘플

### 2.3 Token Parameterization

계층적 구조를 보존하는 여러 방식 비교:

| 기법 | 수식 |
|------|------|
| Trigram | K²c₁ + Kc₂ + c₃ |
| All bigrams | K²(i-1) + Kc_i + c_{i+1} |
| **Prefix-ngram** | **Σₜ₌₁^i K^(i-t)(c_t+1) - 1 (모든 가능한 프리픽스)** |

**Prefix-ngram 동작 방식**: Semantic ID의 모든 프리픽스를 별도 토큰으로 만들어 임베딩 테이블에 lookup 후 합 풀링

```
Semantic ID: (c₁, c₂, c₃, c₄, c₅, c₆)

생성되는 토큰들:
  token₁ = f(c₁)                     ← 1-gram
  token₂ = f(c₁, c₂)                ← 2-gram prefix
  token₃ = f(c₁, c₂, c₃)           ← 3-gram prefix
  token₄ = f(c₁, c₂, c₃, c₄)      ← 4-gram prefix
  token₅ = f(c₁, c₂, c₃, c₄, c₅)  ← 5-gram prefix
     
(token 은 결국 특정 임베딩 idx 인 것이고 DLRM 임베딩 테이블에서 lookup)
                  │
                  ▼
            합 풀링 (sum pooling)
                  │
                  ▼
            최종 아이템 임베딩
```

**새 아이템 cold-start 해결 메커니즘**:

```
[Random Hashing]                    [Semantic ID Prefix-ngram]

신상품 X 등장                        신상품 X 등장
    │                                    │
    ▼                                    ▼
무작위 해시 슬롯 배정               RQ-VAE → c = (3, 127, 44, ...)
    │                                    │
    ▼                                    ▼
랜덤 초기화 임베딩 사용             의미론적으로 유사한 기존 아이템들과
(아무 정보 없음)                    token₁~tokenN 공유
                                         │
                                         ▼
                                    이미 학습된 가중치로 시작
                                    → cold-start 문제 완화 (-0.41% NE)
```

### 2.4 왜 Semantic ID를 그대로 쓰지 않고 Token으로 변환하는가?

핵심 이유: **RQ-VAE의 코드(cᵢ)는 "잔차 기반"이라 앞 코드 없이는 의미가 성립하지 않음**

**옵션 A: 전체 Semantic ID 하나를 단일 토큰으로 사용**

```
(c₁, c₂, c₃, c₄, c₅, c₆) → 단일 슬롯 매핑

문제:
  - 가능한 고유 ID: 2048⁶ ≈ 7.4×10¹⁹ → mod H로 축소 필수
  - (3,127,44,201,8,99)와 (3,127,44,201,8,12)는 거의 같은 아이템인데
    완전 다른 슬롯에 배정됨
  → Random Hashing과 동일한 문제 (지식 공유 불가)
```

**옵션 B: 각 cᵢ를 독립 토큰으로 사용 (L개 토큰)**

```
token_1 = c₁, token_2 = c₂, ..., token_6 = c₆

문제: cᵢ는 앞 코드(c₁,...,c_{i-1}) 기반의 잔차라서 단독으로는 의미 모호

예) c₃ = 44인 두 아이템:
  (c₁=3, c₂=127, c₃=44) → "러닝화 > 경량쿠셔닝형"
  (c₁=9, c₂=500, c₃=44) → "립스틱 > 매트타입"
  
  같은 c₃=44지만 의미 완전 다름
  → c₃만으로 슬롯 공유하면 무관한 아이템끼리 섞임
```

**옵션 C (채택): Prefix-ngram**

각 prefix는 그 자체로 완결된 의미 단위가 되어, 계층 구조를 임베딩 공유 구조로 변환

```
아이템 A: (3, 127, 44, 201, 8, 99)   ← 러닝화 신상
아이템 B: (3, 127, 44, 201, 8, 12)   ← 매우 유사한 러닝화
아이템 C: (3, 127, 44, 555, 7,  3)   ← 같은 세부 카테고리, 다른 모델
아이템 D: (3, 900, 10,  55, 2,  1)   ← 같은 대분류, 다른 종류

Prefix-ngram 공유 패턴:
  A ↔ B : token₁~₅ 공유 (거의 같은 제품군)
  A ↔ C : token₁~₃ 공유 (같은 세부 카테고리)
  A ↔ D : token₁   공유 (같은 대분류)

→ 의미적 유사도에 비례한 다중 granularity 지식 공유
```

**한 줄 요약**: Semantic ID는 계층 구조를 갖지만 단일 식별자일 뿐. Prefix-ngram이 그 계층을 실제 **임베딩 테이블 공유 구조**로 변환하는 장치

## 3. Item Impression Distribution Issues

문제 정의에서 언급한 3가지 이슈를 데이터로 구체화:

**Item Cardinality**: 수십억 개 아이템(I) ≫ 임베딩 슬롯(H) → 충돌 불가피, 충돌 계수 a = I/H ≈ 3

**Impression Skew**:
```
Head  (상위 0.1%)  → 전체 노출의 25%   (소수가 대부분 차지)
Torso (다음 5.5%)  → 전체 노출의 50%
Tail  (나머지 94.4%) → 전체 노출의 25%  (대다수가 학습 기회 부족)
```

**ID Drifting**:
```
Day 0: 코퍼스 C₀
Day 6: C₀의 50%가 시스템 퇴출, 동시에 새 아이템 대거 진입
  → Random Hashing: 새 아이템이 기존 슬롯 덮어쓰기 → 기존 아이템 표현 오염
  → Semantic ID: 유사 아이템끼리 슬롯 공유 → 새 아이템 진입해도 표현 안정
```

## 4. 실험 결과

### 4.0 실험 데이터 및 설정

- **데이터**: Meta 광고 랭킹 데이터
- **훈련**: 4일치 순차 처리, 1 에포크
- **평가**: 훈련 다음날 처음 6시간

### 4.1 평가 지표: Normalized Entropy (NE)

```
        -1/N Σ [yᵢlog(pᵢ) + (1-yᵢ)log(1-pᵢ)]
NE =  ──────────────────────────────────────────
         -(p·log(p) + (1-p)·log(1-p))
```

- 분자: 실제 모델의 cross-entropy loss
- 분모: 항상 평균 CTR(p)로만 예측하는 baseline의 loss
- **낮을수록 좋음**

훈련 데이터와 평가 데이터의 평균 CTR(p)이 다를 수 있어서, 단순 loss 비교 시 모델 성능 차이인지 데이터 CTR 분포 차이인지 구분이 안 됨. NE는 각각의 p로 정규화하여 CTR 분포 차이를 제거하고 순수 모델 성능만 비교

> Meta 기준: NE 개선 > 0.02% 이면 유의미, 본 논문의 온라인 개선 0.15%는 매우 큰 수치

### 4.2 토큰 매개변수화 비교 (4일 훈련)

| 방법 | NE 개선 |
|------|---------|
| Trigram (K=2048, L=3) | -0.028% |
| Prefix-3gram | -0.141% |
| Prefix-5gram | -0.208% |
| **Prefix-6gram** | **-0.215%** ✓ |

### 4.3 아이템 세그먼트별 성능

| 세그먼트 | SemID vs RH |
|---------|-------------|
| Head (상위 0.1%) | 0.00% |
| Tail | -0.04% |
| **새로운 아이템** | **-0.41%** ✓ |
| 전체 | -0.05% |

→ 새 아이템에서 가장 큰 이득: 의미론적으로 유사한 기존 아이템의 가중치 활용

42시간 간격 NE 차이로 분포 이동 안정성도 함께 측정 (낮을수록 안정):
- Random Hashing: 0.0083
- **Semantic ID: 0.0073** ✓ (개별 임베딩 수준의 안정성)

### 4.4 임베딩 표현 공간 분석

같은 슬롯을 공유하는 아이템들의 임베딩 품질을 비교 (충돌 계수 5 기준):

| 방법 | 클러스터 크기 | 내부 분산 | 쌍 거리 |
|------|------------|---------|--------|
| Random Hashing | 5 (평균) | 1.52×10⁻³ | 0.22 |
| SemID (작은 클러스터) | 4~10 | 1.31×10⁻³ | 0.24 |
| **SemID (상위 1,000 클러스터)** | 수천 | **1.23×10⁻³** | **0.06** ✓ |

- **내부 분산**: 같은 슬롯을 공유하는 아이템들의 임베딩이 얼마나 유사한지 → SemID가 낮음 = 의미적으로 유사한 것끼리 모임
- **쌍 거리**: 상위 1,000개 클러스터에서 0.06으로 Random Hashing(0.22) 대비 매우 낮음 → 자주 쓰이는 슬롯일수록 의미 응집도가 높음

### 4.5 사용자 히스토리 모델링

Transformer/PMA 기반 집계 모듈에서 SemID 이득 증폭:
- Bypass: -0.085%
- **Transformer: -0.110%** ✓

Attention 분석: SemID 모델은 패딩 토큰 집중도 9% 감소, Attention 엔트로피 8.5% 감소 → 더 의미 있는 표현 학습

## 5. 프로덕션

### 5.1 훈련 및 서빙 파이프라인

**오프라인 훈련 (RQ-VAE):**
- CU 모델: CC100 데이터 사전학습 + 내부 광고 데이터 fine-tuning
- RQ-VAE: L=6, K=2048, H=O(50M), Prefix-5gram
  - 실제 임베딩 테이블은 5천만개
- 훈련 데이터: 최근 3개월 광고 샘플

```
[의미 표현력]        [표현 공간 용량]        [실제 저장 공간]
   L=6,  K=2048  →  2048⁶ 조합 가능  →  mod H=50M 으로 축소
   (세밀함 조절)    (잠재적 표현력)       (메모리 한계)
```

**온라인 서빙 파이프라인 (Figure 4):**

```
[광고 생성 시점]
  새 광고 등록
      │
      ▼
  CU 모델 → 콘텐츠 임베딩 z
      │
      ▼
  RQ-VAE 인코딩 → Semantic ID (c₁, ..., c₆)
      │
      ▼
  Entity Store (사전 저장)
      │
      │          [랭킹 요청 시점]
      │
      ▼
  Feature 생성 단계
  ┌─────────────────────────────────────┐
  │  Target 아이템 raw ID               │
  │      → Entity Store에서 SemID 조회  │
  │  User History raw ID 목록           │
  │      → Entity Store에서 SemID 조회  │
  └──────────────────┬──────────────────┘
                     │
                     ▼
            Semantic Feature 생성
                     │
                     ▼
            DLRM 랭킹 모델 입력
```

핵심: Semantic ID는 광고 생성 시점에 **미리 계산되어 저장**되므로 랭킹 요청 시 추가 추론 비용 없음

### 5.2 결과

**Meta 광고 랭킹 시스템** (수십억 사용자 규모):

| 지표 | 값 |
|------|-----|
| 온라인 최상위 메트릭 개선 | **+0.15%** |
| A/A 분산 감소 (예측 안정성) | **43% 감소** |

- NE 이득 > 0.02%를 Meta에서 "유의미"로 간주 → 0.15%는 매우 큰 개선
- A/A 분산 감소의 대부분은 tail 아이템에서 발생

**프리픽스 깊이별 의미-예측 유사성**: 프리픽스가 깊을수록 동일 클러스터 내 아이템 간 CTR 손실률(CLR) 단조 감소 → 계층 구조가 실제 의미 유사성 반영

---

## 6. 결론 및 의의

- **Semantic ID Prefix-ngram**이 임베딩 표현 불안정성의 3가지 근본 원인(cardinality, skew, drift)을 동시에 완화
- 특히 **새 아이템과 tail 아이템** 일반화에서 유의미한 성능 향상
- 계층적 클러스터링 구조를 토큰 매개변수화에 직접 반영하는 방식이 핵심
- 대규모 온라인 광고 시스템에서 실제 배포 검증 완료
